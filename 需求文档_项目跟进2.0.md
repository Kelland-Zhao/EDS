## 1. 项目背景

项目跟进模块用于追踪部门项目进展。项目分两类，数据形态不同：

| 类型 | 说明 | 节点形态 |
|------|------|----------|
| **标准（standard）** | 新品 / 新自动化项目 | 标准 milestone（模具改造、FAT、现场安装、IQ/OQ、工程测试、PQ、Mass Production…），有先后阶段 |
| **CI** | 部门持续改进 | 无标准模板的 follow-up 事项，重点是「责任人 + 完成时间」，事项间无固定先后 |

当前模块只支持标准类型，本次扩展为**同一张表 / 同一页面**同时支持两类，并按类型自适应录入表单与可视化样式。

> **范围边界**：本次 scope = **项目录入 + 可视化**。
> 定时到期 Gmail 提醒（提醒 Leader 及其 line manager）**不在本次范围**，仅在数据结构上为其预留（每节点记录 owner 及邮箱）。

---

## 2. 核心功能

### 2.1 录入
- 新增项目时**先选类型**（标准 / CI）。
- **标准**：里程碑模板 + 可动态增减节点；每节点填「计划日期」；责任人默认= 项目 Leader（节点不单独填 owner）。
- **CI**：逐条添加 follow-up 事项，每条填「事项名称 + 责任人（从用户权限表选人）+ 计划完成日期」。
- 编辑：实际完成日期可后续填写；管理员可改状态 / 计划日期 / 节点结构（沿用现有权限）。

### 2.2 可视化（类型自适应）
- **主列表（两类共用一行）**：项目名称 | 类型徽章 | Leader | 状态 | 进度+下一个到期（X/Y 完成、最近未完成节点日期、逾期红标）。支持按类型筛选。
- **展开详情**：
  - 标准 → 里程碑时间线（计划 vs 实际，保留现有样式）。
  - CI → 清单表：`事项 | 责任人 | 计划完成 | 实际完成 | 状态`。
- 两类底层同源 `milestones` JSON，仅前端渲染分支不同，**无数据冗余**。

## 3. 数据源 / 数据模型

### 3.1 `项目总表` 列结构（在现有 5 列后追加 1 列，0-indexed）

| 列 | index | 字段 | 说明 |
|----|-------|------|------|
| A | 0 | 项目名称 | |
| B | 1 | 项目 Leader | 含邮箱，如 `郭彭 (andy_guo@colpal.com)` |
| C | 2 | 测试责任技术员 | |
| D | 3 | 状态 / Status | Not start / Ongoing / Done |
| E | 4 | Milestones_JSON | 单一数据源，见 3.2 |
| **F** | **5** | **类型 / Type** | **新增**：`标准` 或 `CI`，缺省按 `标准` |

> 选择「追加到 F 列」而非插入，避免移动 E 列 JSON 的索引，最小改动。
> 对应常量新增 `PROJECT_TYPE_COL = 5`。

### 3.2 Milestones_JSON 节点结构（扩展 owner）

```json
[
  {
    "name": "事项/里程碑名称",
    "planned": "2026-06-15 | NA",
    "actual": "2026-06-18 | ''",
    "owner": "王鑫",                       // 新增；标准项目可空(=项目Leader)
    "ownerEmail": "xin_wang@colpal.com"     // 新增；CI 选人时一并写入，为将来提醒预留
  }
]
```

- 标准项目：`owner`/`ownerEmail` 可省略，渲染与提醒时回退到项目 Leader。
- CI 项目：每条事项必须有 `owner`（从用户权限表选人，自动带出 `ownerEmail`）。
- 「完成」判定：`actual` 非空即完成。

### 3.3 关联数据
- 用户权限表（`USER_PERMISSION_SS_ID` / `USER_PERMISSION_SHEET_NAME`）：
  - 姓名 = index 1，邮箱 = index 9，直线上级邮箱 = index 60（BI 列，为将来 line manager 提醒预留）。
- 历史表 `ProjectTracking_History`、审批表 `ProjectTracking_Approvals`：沿用，按节点 name 记录。

## 6. 权限 / 角色（沿用现有）
- 管理员（权限表 BH 列 = `管理员`）：改状态、计划日期、节点结构。
- 普通用户：填写实际完成日期。
- 计划日期推迟：走现有上级审批流（`requestPlannedDateDelay` → `handleApprovalAction`），已改为按项目自身 JSON 节点取数。
- CI 实际完成更新：沿用「普通用户可改 actual」。（待确认：是否限定仅本人 owner 可改本人事项 → 见第 9 节）

---

## 7. 边界条件
- 历史 10 个项目均为标准类型 → F 列回填 `标准`。
- Type 缺省/空 → 按 `标准` 处理（向后兼容）。
- 一个项目只能属于一种类型，不混用。
- CI 录入时责任人必填；选人不在权限表中 → 不允许（保证可查邮箱）。
- 节点 `actual` 非空 = 完成；`planned=NA` 表示无计划日期，不参与逾期判断。
- 逾期判断（仅可视化红标）：`actual` 为空且 `planned` 为有效日期且 < 今天。

---

## 8. 验收标准
1. 新增标准项目：选「标准」→ 里程碑模板录入 → 列表显示类型徽章「标准」，展开为时间线。
2. 新增 CI 项目：选「CI」→ 逐条录入事项+责任人(选人)+计划完成 → 列表显示「CI」，展开为清单表含责任人列。
3. CI 责任人从权限表选人后，JSON 内正确写入 `owner` + `ownerEmail`。
4. 主列表「进度 + 下一个到期 + 逾期红标」两类均正确。
5. 实际完成日期填写后，节点状态变「完成」，进度更新。
6. 历史标准项目不受影响，类型显示「标准」，时间线展示正常。
7. 管理员改计划日期 / 延期审批通过后，前端（读 JSON）正确反映。
8. 全程单一数据源（E 列 JSON + F 列 Type），无双写、无冗余列。

## 9. 决议（已确认）
1. **CI 实际完成的编辑权限**：✅ **任何登录用户可改**（沿用现有「普通用户可改 actual」模型，与标准项目一致）。
2. **CI 项目级 Leader**：✅ **仍必填**（作为项目总负责人/汇总联系人；事项责任人在节点级单独指定）。
3. **标准项目节点单独 owner**：本期**不做**，默认 = 项目 Leader，保持简单。
4. **按类型分组统计**：本期**仅做类型筛选**，不做分组/待办总数统计。

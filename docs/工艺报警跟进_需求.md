# 需求文档：EDS 新增"注塑工艺报警跟进"模块

## 1. 项目背景
高露洁三笑设备部使用 IOT/Opera 系统自动采集注塑工序的工艺报警数据（温控箱温度、成型周期、炮筒温度、漏料、冷却水等），写入独立的 Google Sheet（`1FtljG58B1uzcgNU_vKEYUSeW39X23ljUYtngEYmWl_A`，标题"IOT/Opera 报警数据新版格式"）。

目前这些报警记录的"原因分析、跟进措施/解决措施、状态、跟进人"等字段为空，需要一线人员（机台操作工/技术员）登录 EDS 后，按自己负责的机台/车间筛选出未处理的报警，逐条补全跟进信息并提交，写回该 Spreadsheet 对应行。

## 2. 核心功能
1. 一线登录 EDS 后进入"工艺报警跟进"新模块
2. 按模块内 Tab 切换 5 类报警（温控箱温度 / 周期 / 温控箱跟从 / 炮筒温度 / 漏料），冷却水单独一个 Tab（无机台号，见 6.2）
3. 每个 Tab 默认按用户登录信息中的 **车间(Workshop: TB1/TB2)** 过滤，并提供 **机台号** 下拉/搜索框进一步筛选
4. 列表默认只显示"未处理"记录（跟进字段为空），可切换显示已处理记录（参考 [FailureReport_Followup_Manage-js.html](FailureReport_Followup_Manage-js.html) 的"已通过/已取消"展开折叠模式）
5. 列表展示报警关键信息（报警编号、时间、机台、参数实际值/标准值/差异等只读），以及可编辑的跟进字段（原因分析、跟进措施/解决措施、状态、跟进人）
6. 逐行"保存"提交，写回 Spreadsheet 对应行（按"报警编号"定位行）

## 3. 数据源与字段映射

Spreadsheet ID: `1FtljG58B1uzcgNU_vKEYUSeW39X23ljUYtngEYmWl_A`

| Tab | Sheet名 | 筛选列 | 一线可编辑字段 |
|---|---|---|---|
| 温控箱温度 | IOT温控箱邮件反馈格式 | 车间/机台号 | 原因分析、跟进措施、状态、跟进人 |
| 成型周期 | IOT/Opera邮件反馈格式 | 车间/机台号 | 原因分析、解决措施、状态、跟进人、备注 |

| 炮筒温度 | Opera炮筒温度邮件反馈格式 | 车间/机台号 | 原因分析、解决措施、状态、跟进人、备注 |

> 各 Sheet 中的"模具组跟进措施"/"跟进完成时间"/"是否发送"字段 **不开放给一线**，留给模具组/系统处理。

## 4. 触发方式
- 在 EDS Navigation 中新增菜单入口"工艺报警跟进"
- 登录用户：所有有 Process=IM（注塑）权限的用户

## 5. 输出渠道
- 直接写回 `1FtljG58B1uzcgNU_vKEYUSeW39X23ljUYtngEYmWl_A` 对应 Sheet 的对应行（按"报警编号"列匹配定位行号后 `setValues` 更新指定列）
- 冷却水 Sheet 无唯一编号列，按"Date+Hour"组合定位行

## 6. 配置项与待确认事项

### 6.1 状态枚举值
当前各 Sheet 的"状态"列大多为空；"IOT/Opera邮件反馈格式"中已存在系统自动写入的"正常/未正常"值。
**提议**：一线填写的"状态"统一使用下拉选项：`处理中 / In Progress`、`已完成 / Completed`。已有"正常/未正常"的行（IOT/Opera Tab）不允许一线修改状态列，仅可填写原因分析/解决措施/跟进人/备注。
- [ ] 请确认此状态枚举及"正常/未正常"行的处理方式是否OK

### 6.2 冷却水 Tab 特殊性
冷却水 Sheet 记录粒度是"日期+小时"，无机台号/车间字段，推测对应某一套公共冷却水系统（非单台机）。
**提议**：冷却水 Tab 不做机台筛选，直接列出最近N天（如7天）内"状态为空"的记录供一线补全。
- [ ] 请确认冷却水记录的归属（是否对应特定车间/系统）以及筛选范围

### 6.3 机台号与用户信息关联
用户登录信息（userID表）含 Workshop(TB1/TB2) 和 Process，但无具体机台号列表。
**提议**：默认按用户 Workshop 过滤列表，机台号通过下拉框（从当前 Sheet 数据中提取去重机台号列表）二次筛选，不做强制权限限制（一线可看到本车间所有未处理机台报警）。
- [ ] 请确认是否需要更严格的机台权限控制（如只能看到自己负责的1-2台机）

### 6.4 跟进人字段
**提议**：跟进人字段提交时自动填入当前登录用户姓名（只读展示，不需手动输入）。
- [ ] 请确认

### 6.5 历史数据量
"IOT/Opera邮件反馈格式" 现有 18000+ 行，"IOT温控箱邮件反馈格式" 6000+ 行。
**提议**：后端按"状态为空"且日期在最近30天内过滤后再返回前端，避免一次性加载全部历史数据。
- [ ] 请确认时间范围（30天 / 其他）

## 7. 邮件推送功能（班次报警汇总）

### 7.1 功能描述
新增定时邮件推送：每天 **08:00 / 16:00 / 00:00** 三个时间点，自动汇总**上一班次**产生的工艺报警（无论是否已填写跟进信息），以邮件形式发送。

### 7.2 班次与时间对应关系
三班制衔接：
| 发送时间 | 汇总班次 | 报警时间范围 |
|---|---|---|
| 08:00 | 夜班（三班） | 00:00–08:00 |
| 16:00 | 早班（一班） | 08:00–16:00 |
| 00:00 | 中班（二班） | 16:00–24:00 |

### 7.3 数据范围
全部 6 类报警 sheet（温控箱温度、周期、温控箱跟从、炮筒温度、漏料、冷却水），按"报警时间"落在上述时间窗口内的记录汇总。每类报警在邮件中分区块展示（区块标题 + 表格），无记录的区块可省略或显示"本班次无报警"。
表格需包含：报警基本信息（机台号/时间/参数实际值与标准值差异等）+ 当前跟进状态（原因分析/跟进措施/状态/跟进人——若已由一线填写则显示，未填写则显示"未跟进/Not Followed Up"）。

### 7.4 收件人 — 待确认
- [ ] 收件人范围、分组方式（按车间/班次/固定列表）待后续确定

### 7.5 样式与发送规范
遵循 [邮件通知_通用需求.md](邮件通知_通用需求.md) 通用规范（中英双语、表头 `#E60012`、HTML富文本、页脚免回复说明）。

### 7.6 触发实现
复用 Spreadsheet 中已有的"定时设置"sheet 机制（`月份/周几/第几周/日期/时间/描述/函数名`），新增3条定时任务对应 08:00/16:00/00:00 触发，函数名指向新增的汇总发送函数（如 `sendShiftAlarmSummaryEmail`）。

## 8. 验收标准
1. 一线登录后能看到本车间各类报警的未处理列表，按机台号可二次筛选
2. 可在列表内编辑原因分析/跟进措施/状态/跟进人/备注并保存
3. 保存后数据正确写回 Spreadsheet 对应 Sheet 对应行，且不影响其他列（特别是"是否发送"等系统列）
4. 已处理记录默认收起，可展开查看
5. 页面在移动端（车间现场常用手机/平板）布局正常
6. 每天 08:00/16:00/00:00 准时收到上一班次报警汇总邮件，内容覆盖6类报警且跟进状态展示正确

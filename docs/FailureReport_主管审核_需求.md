# 故障报告主管审核 — 需求文档

## 项目背景

当前故障报告提交后直接进入 PA 跟进 → 验证 → 结案流程，缺少直线上级对报告内容（RCA 分析、CAPA 措施）的审核环节。需要插入主管审核步骤，审核通过后才能进入后续流程。

## 核心流程

```
责任人提交报告 → 主管审核 → 通过 → 生成PDF + 创建PA跟进 → 验证 → 结案
                         ↘ 退回 → 责任人修改 → 重新提交
```

### 与现有流程对比

| 环节 | 修改前 | 修改后 |
|---|---|---|
| 提交报告 | 生成 PDF + 创建跟进 + 发邮件 | **仅保存数据，状态=主管审核中，邮件通知主管，抄送责任人** |
| 主管审核 | 不存在 | **新增：审核通过 / 退回修改** |
| 审核通过 | — | **触发 PDF 生成 + 创建跟进 + 通知 PA 责任人（抄送主管）** |
| 退回修改 | — | **报告退回 Upload 列表，邮件通知责任人（抄送主管），责任人可重新编辑提交** |

## 权限数据与工序映射（实测，必读）

### 工序代码归一化

两张表工序代码不一致，路由匹配前必须归一：

| 表 | 列 | 代码 |
|---|---|---|
| Failure_Database | F (5) | `IM` / `TF` / `PK` |
| userID | O (14) | `INJ` / `TF` / `PK` |

> **归一规则**：`IM → INJ`，`TF → TF`，`PK → PK`。仅 IM↔INJ 不同。匹配工序管理员/统计时统一转成 userID 侧代码（INJ/TF/PK）。

### 故障报告管理员（按工序，BF 列 = `Y`）

userID 表 **BF 列（索引 57，「故障报告管理权限」）= `Y`** 标记该工序管理员，每工序一人（实测）：

| 工序(O列) | 管理员 | 邮箱(J列) |
|---|---|---|
| INJ (=IM) | 赵阳 | kelland_zhao@colpal.com |
| TF | 贾海龙 | david_jia@colpal.com |
| PK | 王鹏 | jack_wang@colpal.com |

> **实现以表为准，不写死姓名/邮箱**：运行时扫描 userID，取 `O列 == 报告工序(归一后) 且 BF列 == 'Y'` 的行，读 J 列邮箱。

> **边界（一个工序匹配到多个/零个管理员）**：
> - **多个 `Y`**：全部作为该工序管理员，通知时**全部抄送/并列收件**，数据范围取并集（任一管理员均可审核本工序报告）。
> - **零个 `Y`**：回退通知**系统管理员**（kelland_zhao@colpal.com），并在日志记录"工序 X 未配置故障报告管理员"，提示补配 BF 列。

### 「管理员」的两种用途（务必区分）

| 用途 | 实现 | 说明 |
|---|---|---|
| **页面访问门控**（能否看到审核页内容） | `checkFailureReportManagePermission(email,name)`（BF=Y，管理员） **OR 邮箱出现在任一 BI 列（主管）** | ⚠️ 不能只判 BF=Y——纯主管（非 BF=Y）也有待审报告，只判 BF 会把主管挡在外面。二者满足其一即放行 |
| **按工序路由 / 数据范围**（发给谁、看哪些） | **新增逻辑**：`O列 == 报告工序(归一) 且 BF == 'Y'` → 工序管理员邮箱 | 各工序报告只发各工序管理员，不广播全部 |

### 直线上级（BI 列）兜底

- BI（索引 60）当前**填充稀疏**，多数责任人为空。
- 提交时若责任人 BI 为空 → 按「无主管」处理：通知**该报告工序的管理员**审核（见无主管流程）。
- 即责任人有 BI → 主管审核；无 BI → 工序管理员代为审核。流程不因 BI 缺失而中断。

### 历史数据迁移

上线时对已有报告（有附件=已提交过的）执行一次性初始化：
- 审核状态 = `已通过`
- 审核日期 = 上传日期（I列）
- 审核人 = 责任人（L列）—— 兜底值
- 退回原因 = 空

## 页面设计

### 1. 主管审核页面（新建）

- 文件名：`FailureReport_Review.html` + `FailureReport_Review-js.html`
- 入口：首页导航栏新增「故障报告审核 / Report Review」
- 权限控制：**前端方案** — 入口对所有用户可见，页面加载时调用后端检查权限，无权限者显示提示信息

#### 后端查询函数

新增 `getPendingReviewReports(userEmail, userName)`（**当前用户身份从前端传入**，参照 Upload 的 `currentUserEmail/currentUserName`，用于匹配 BI 主管和 BF 管理员）：
- 查询 Failure_Database，筛选审核状态 = `主管审核中` 或 `已退回`
- **管理员**（BF=Y）：返回**本人所属工序**（其 O 列工序，归一后）的主管审核中/已退回报告，**非全部**
- **主管**（邮箱出现在某行 BI）：返回直线上级匹配当前用户的报告（userID 表 BI 列 = 当前用户邮箱 → 该行 NAME 列 = 报告责任人）
  - ⚠️ 责任人（L 列）存储格式为 `姓名【邮箱】`，匹配 NAME 前需先 `extractName()` 提取姓名
- ⚠️ **若用户同时是主管和工序管理员，返回二者并集**（其下属报告 ∪ 本工序全部待审报告），去重
- 返回字段：报告编号、机台号、问题描述、工序、责任人、提交日期、审核状态、内容（JSON，用于详情弹窗）
- ⚠️ **查看详情依赖 K 列 JSON**：仅新提交报告有 JSON，老数据（K 列空）无法回显——但老报告均已迁移为「已通过」、不进审核页，故不受影响

#### 页面布局
- 表格列：报告编号、机台号、问题描述、责任人、提交日期、审核状态
- 筛选：按工序（IM/TF/PK）
- **默认展示**：主管审核中 + 已退回 的报告（已通过及后续状态在 Progress 页面展示）
- 操作列：
  - `查看详情 / View Detail` — 弹窗展示完整 BDA 表单内容（只读），复用 Template 表单组件
  - `通过 / Approve` — **SweetAlert 二次确认** → 确认通过
  - `退回 / Return` — **SweetAlert 弹窗**填写退回原因（非必填）→ 退回给责任人
- 操作按钮样式：`display:block;width:120px;margin:2px auto;`（同 Upload 页面风格）
- 状态列颜色：
  - 主管审核中 = 黄色
  - 已通过 = 绿色
  - 已退回 = 红色

### 2. Upload 页面改造

- ⚠️ **可填写/可修改判定改为按「审核状态」，不再依据附件(J列)/uploadDate**：
  - 现状（`FailureReport_Upload-js.html`）：「状态」徽章键于 **`attachments`(J列)**（有→已上传/无→未上传），且「填写故障报告」按钮**无条件渲染**（每行都显示）。
  - 改造后 submit 不再写附件(J列)/uploadDate（PDF/上传移到审核通过），「主管审核中」与「未提交」的 attachments 同为空、无法区分。若沿用 attachments 判定，责任人会在审核期间重复提交，**违反边界条件#3**。
- ⚠️ **「状态」列由审核状态接管**（取代现有 attachments 驱动的已上传/未上传徽章），**「填写」按钮由无条件改为按审核状态条件渲染**：

| 审核状态 | 状态列标记 | 按钮 |
|---|---|---|
| 空（从未提交） | "未提交" | 「填写故障报告」→ 进 Template 新建 |
| 已退回 | "已退回"（红色）+ 显示退回原因 | 「修改 / Edit」→ 进 Template 回填编辑 |
| 主管审核中 | "审核中"（灰色） | **隐藏/置灰，不可操作**（防重复提交） |
| 已通过及之后 | 不在 Upload 出现（去 Progress） | — |

- 已退回的报告：点「修改 / Edit」进入 Template 编辑模式
- **Template 回填**：复用现有 `editMode=1` 入口从 K 列 JSON 回填表单
- ⚠️ **与现有 editMode 的关键区别（必须实现）**：现有 `editMode=1`（Progress 页「修改」触发，见 `submitFailureReport` 的 `isEditMode` 分支）**会立即重新生成 PDF + 同步跟进记录**。退回-修改**不能复用这套逻辑**，否则会绕过主管审核直接出 PDF/跟进，审核形同虚设。
  - 退回重提的正确行为：**仅保存 JSON（追加 reviewHistory）→ 审核状态回「主管审核中」→ 清空退回原因 → 通知主管**，**不生成 PDF、不创建/更新跟进**。
  - 实现方式：用**审核状态**区分行为分支（当前状态=已退回 → 走"仅保存+回审核中"路径），而非新增第二个近义参数；避免 `editMode=1` 与 `mode=edit` 并存造成混淆。

### 3. Progress 页面改造

- **审核未通过的报告不进入 Progress 页面**：仅排除审核状态 = `主管审核中` / `已退回` 的报告
- ⚠️ **不能简单写成"仅已通过才显示"**：从未提交的报告（uploadDate 空、审核状态为空）现状在 Progress 显示为"未上传"，若改成"仅已通过显示"会让这批报告**从 Progress 消失（回归）**。
  - 正确规则：审核状态 ∈ {主管审核中, 已退回} → 不显示；其余（空 / 已通过 / 已上传 / 已完成）→ **按现有派生逻辑照常显示**
- 现有派生逻辑不变（无附件→未上传；有附件+跟进全过→已完成）

> ⚠️ **过滤必须按页面区分，不能改公共后端函数**：Upload（`getFailureReportProgressData(email,name)`）与 Progress（`getFailureReportProgressData()`）**调用同一个后端函数**。若在该函数内部过滤掉「主管审核中/已退回」，会同时把 Upload 需要的「已退回/未提交」滤没。
> 正确做法：
> - 公共函数**新增返回 O–R 字段**（审核状态、退回原因等），不在函数内按状态过滤；
> - 各页面**前端按审核状态各自过滤**（Progress 排除主管审核中/已退回；Upload 只留 空/已退回）。

## 后端逻辑

### 提交逻辑改造 (submitFailureReport)

```
修改前：
  保存JSON → 生成PDF → 更新附件+上传日期 → 创建跟进 → 通知PA责任人

修改后（方案B：提交时暂存 PDF）：
  生成 PDF（照片此刻在手）→ 存入 Drive，URL 记入 K 列 JSON 的 _pdfUrl/_pdfFileName（暂存，不写 J 列）
  → 保存JSON（含 reviewHistory + 暂存PDF信息）→ 写入审核状态=主管审核中 + 清空退回原因(R列)
  → **不写 uploadDate(I列)、不写附件(J列)、不建跟进**（落地移到审核通过执行）
  → 查询 userID 表获取责任人直线上级（BI列）
  → 无直线上级：通知该报告工序的管理员处理（O列==工序 且 BF==Y）
  → 有直线上级：通知主管审核（邮件抄送责任人）

> **为何提交时就生成 PDF（方案B）**：PDF 需嵌入设备照片 `data.photo`，而照片只在提交时在手、且存进 K 列 JSON 会超单元格上限。故提交时即生成 PDF 暂存（业务上仍是"主管审核中"，J列/跟进未落地、Upload/Progress 不显示），审核通过只做"落地"，不重新生成 → 照片不丢、不碰单元格上限。
```

> ⚠️ **改造范围必须覆盖 `submitFailureReport` 内两个分支**：当前函数同时服务"新建提交"和"editMode 修改"（`isEditMode` 分支，Progress 页「修改」触发），**两个分支都会生成 PDF + 建/同步跟进**。若只改新建分支，Progress 页「修改」仍会绕过审核直接出 PDF/跟进。
> 处理原则：
> - 新建提交 / 退回重提 → 走"仅保存 + 主管审核中"路径，**不出 PDF/跟进**
> - 已通过报告的后续编辑（Progress 页 editMode）→ 维持现有"重生成 PDF + 同步跟进"行为（已审核通过，无需再审）
> - 用**审核状态**判定走哪条路径，避免误伤。

### 审核通过逻辑 (approveReport)

```
  ① 并发校验：重新读取当前审核状态，若 ≠ 主管审核中 → 返回"该报告已被处理"，终止
     （防两个管理员同时点通过导致重复建跟进）
  ② 幂等：先按 case_code 删除本报告已有跟进记录（防重试重复）
  ③ 从 K 列 JSON 读取暂存 PDF（_pdfUrl）→ 落地写附件(J列)+上传日期(I列)
     （不重新生成 PDF；若 _pdfUrl 缺失 → 返回错误提示责任人重新提交）
  ④ 创建PA跟进记录（状态=待验证，附件链接列写 _pdfUrl）
  ⑤ 更新审核状态=已通过 + 审核人 + 审核日期 → 追加 reviewHistory 到 K 列 JSON
  ⑥ 邮件通知PA责任人（复用 sendFollowupCreationReminderEmails）
```

> **顺序要点**：先出 PDF + 建跟进，全部成功后再落「已通过」状态。
> 若 PDF 生成失败 → 见下方容错：**不建跟进、不置已通过**，保持「主管审核中」并提示重试，避免出现"已通过但无附件、跟进链接为空"的脏数据。

### 审核退回逻辑 (returnReport)

```
  更新审核状态=已退回 → 更新审核人+审核日期+退回原因
  → 追加 reviewHistory 到 K 列 JSON
  → 邮件通知责任人修改（含退回原因，抄送主管）
  → 报告重新出现在 Upload 列表
```

### PDF 生成失败容错

方案B 下 **PDF 在提交时生成**（非审核通过时）：
- 提交时若 PDF 生成失败 → submit 整体抛错，报告保持原状态（不置主管审核中），责任人重新提交即可。
- 审核通过(approveReport)只读取暂存 PDF 落地、不重新生成，故**审核环节不会因 PDF 生成失败**。
- 兜底：若审核时发现 _pdfUrl 缺失（异常情况）→ 返回"缺少暂存 PDF，请责任人重新提交"，状态保持主管审核中。

> ⚠️ **重试幂等性（必须实现）**：approveReport 步骤 ② 已建跟进、步骤 ③ 写状态前若中断，重试会重复生成 PDF + 重复建跟进。因此 approveReport **必须做成可重入**：
> - 进入 ② 前，**先按 case_code 删除/覆盖本报告已有跟进记录再重建**（复用现有 editMode 分支的 paIndex 去重/upsert 逻辑），不要无脑追加；
> - 旧 PDF 若已生成，重试时按 editMode 同款逻辑删除旧文件再建，避免 Drive 残留。

## 状态流转

### 报告级别状态

```
主管审核中 / Under Review      — 提交后等待主管审核
已通过 / Approved             — 主管审核通过（进入跟进流程）
已退回 / Returned             — 主管退回修改
已上传 / Uploaded             — PDF已生成，跟进进行中（同现有）
已完成 / Completed            — 所有跟进项已通过（同现有）
```

### 报告状态机

```
                        ┌── 通过 ──→ 已通过(生成PDF+跟进) ──→ 跟进流程...
主管审核中 ──┤
                        └── 退回 ──→ 已退回 ──→ 责任人修改 ──→ 重新提交 ──→ 主管审核中
```

### Progress 页面可见性

```
审核状态 = 主管审核中 / 已退回    →  不显示
审核状态 = 空 / 已通过 / 其它      →  显示（按现有派生逻辑：无附件→未上传，有附件→已上传/已完成）
```

> 注意：历史迁移只给"有附件"的报告设「已通过」，从未提交的报告审核状态为空——这批必须仍显示为"未上传"，故按"排除主管审核中/已退回"而非"仅显示已通过"来过滤。

| 角色 | 权限 |
|---|---|
| 主管 | 审核其下属的主管审核中报告（userID BI 列匹配） |
| 责任人 | 查看自己被退回的报告，修改并重新提交 |
| 管理员（按工序） | 审核**本工序**报告（O列==工序 且 BF==Y）+ 处理本工序无主管的报告 |
| 验证人/PA责任人 | 仅在审核通过后参与后续流程 |

### 主管身份判定
- userID 表中 `直线上级邮件` 列（BI 列，索引 60）
- 当前登录用户的邮箱出现在该列中 → 该用户是主管，可审核对应下属
- 单个主管（BI 列存单个邮箱）

### 导航入口可见性（前端方案）
- 审核页面入口对所有用户可见
- 页面加载时调用后端权限检查：**BF=Y（管理员）OR 邮箱出现在任一 BI 列（主管）** → 有权限
- 无权限者：页面显示 "您暂无审核权限 / No review permission" 提示

## 边界条件

1. **退回后修改**：责任人只能修改自己提交的、状态为「已退回」的报告
2. **重复提交**：已审核通过的报告不可再修改（同现有逻辑：有附件=不可修改）
3. **审核中修改**：主管审核期间（状态=主管审核中），责任人不可修改报告
4. **审核超时**：暂不设超时，后续迭代可加提醒
5. **无主管情况**：提交时若责任人无直线上级 → 邮件通知**该报告工序的管理员**（O列==工序 且 BF==Y；零个则系统管理员）手动处理（不自动通过）
6. **Template 回填**：退回修改时从 K 列 JSON 读取表单数据回填，重新提交追加 reviewHistory
7. **修改历史**：审核记录追加到 JSON 的 `reviewHistory` 数组，不覆盖，便于追溯
8. **退回重提后审核人不变**：重新提交后直线上级不变，审核主管为同一人
9. **管理员代审**：管理员审核非自己下属的报告时，审核人字段记录管理员本人
10. **退回次数不限**：允许无限次退回-修改循环，第一版不设限制

## 交互细节

- **通过操作**：点击「通过」→ SweetAlert 确认弹窗 "确认通过此报告？" → 确认后执行
- **退回操作**：点击「退回」→ SweetAlert 输入弹窗（textarea，非必填）→ 确认后执行
- **查看详情**：弹窗展示 BDA 表单（只读模式），复用 Template 页面表单组件

## 验收标准

- [ ] 责任人提交报告后，状态为「主管审核中」，PDF 未生成，跟进未创建
- [ ] 主管收到审核通知邮件（抄送责任人）
- [ ] 主管在审核页面看到主管审核中报告列表（按工序可筛选）
- [ ] 主管可查看报告详情（BDA 只读弹窗）
- [ ] 主管可通过（二次确认）/ 退回（可填退回原因）
- [ ] 通过后自动生成 PDF + 创建 PA 跟进，通知 PA 责任人（抄送主管）
- [ ] 退回后报告回到 Upload 列表，责任人可修改重提，退回原因可见
- [ ] 退回后责任人收到通知邮件（含退回原因，抄送主管）
- [ ] 审核未通过（主管审核中/已退回）的报告不显示在 Progress 页面
- [ ] **从未提交的报告（审核状态为空）仍在 Progress 显示为"未上传"（无回归）**
- [ ] 无直线上级时通知**该报告工序的管理员**（非广播全部管理员）
- [ ] 管理员审核页只看到**本工序**的待审报告
- [ ] **退回重提不生成 PDF、不创建跟进，状态回到「主管审核中」**（不绕过审核）
- [ ] **「主管审核中」报告在 Upload 不可填写/修改**（审核期间禁止责任人重复提交）
- [ ] **纯主管（非 BF=Y）也能进入审核页**（门控覆盖 BI 匹配，不只判 BF）
- [ ] **Upload 与 Progress 共用后端函数，过滤互不影响**（Upload 仍见已退回/未提交，Progress 仍排除主管审核中/已退回）
- [ ] **PDF 生成失败时整体回滚，状态保持「主管审核中」，可重试**
- [ ] **approveReport 重试不产生重复 PDF / 重复跟进记录**（可重入幂等）
- [ ] 历史报告自动迁移为「已通过」状态
- [ ] 审核记录写入 JSON reviewHistory，可追溯
- [ ] 导航入口前端方案：入口常显，无权限者进入后看到提示

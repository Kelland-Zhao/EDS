# 故障报告主管审核 — 需求文档

## 项目背景

当前故障报告提交后直接进入 PA 跟进 → 验证 → 结案流程，缺少直线上级对报告内容（RCA 分析、CAPA 措施）的审核环节。需要插入主管审核步骤，审核通过后才能进入后续流程。

## 核心流程

```
责任人提交报告 → 主管审核 → 通过 → 生成PDF + 创建PA跟进 → 验证 → 结案
                         ↘ 退回 → 责任人修改 → 重新提交
```

### 与现有流程对比

| 环节 | 修改前 | 修改后 |
|---|---|---|
| 提交报告 | 生成 PDF + 创建跟进 + 发邮件 | **仅保存数据，状态=主管审核中，邮件通知主管，抄送责任人** |
| 主管审核 | 不存在 | **新增：审核通过 / 退回修改** |
| 审核通过 | — | **触发 PDF 生成 + 创建跟进 + 通知 PA 责任人（抄送主管）** |
| 退回修改 | — | **报告退回 Upload 列表，邮件通知责任人（抄送主管），责任人可重新编辑提交** |

## 数据库现状

### Failure_Database（14列，A-N）

| 列 | 字段 |
|---|---|
| A | 编号 |
| B | 机台号 |
| C | 问题描述 |
| D | 提交日期 |
| E | 车间 |
| F | 工序 |
| G | 故障报告编号 |
| H | 分配日期 |
| I | 上传日期 |
| J | 附件 |
| K | 内容（JSON） |
| L | 责任人 |
| M | 完成天数 |
| N | 维修时间 / Repair Time |

### 新增列（O-R，15-18）

| 列号 | 字段名 | 说明 |
|---|---|---|
| O (15) | 审核状态 / Review Status | 主管审核中 / 已通过 / 已退回 |
| P (16) | 审核人 / Reviewed By | 审核人姓名【邮箱】 |
| Q (17) | 审核日期 / Review Date | yyyy-MM-dd |
| R (18) | 退回原因 / Return Reason | 退回时填写（非必填），重新提交后清空 |

> 列 15-18（O-R）当前未被占用，可直接追加。
> 直线上级从 userID 表 BI 列读取，不单独存入 Failure_Database。

### 修改历史（存入 JSON）

为支持追溯，审核记录追加到第 K 列 JSON 的 `reviewHistory` 数组，而非覆盖：

```json
{
  "reviewHistory": [
    {
      "action": "submitted",
      "timestamp": "2026-06-13 14:30:00",
      "operator": "张三 / Zhang San"
    },
    {
      "action": "returned",
      "timestamp": "2026-06-13 15:00:00",
      "operator": "李四 / Li Si",
      "reason": "RCA分析不够深入，请补充"
    },
    {
      "action": "approved",
      "timestamp": "2026-06-14 09:00:00",
      "operator": "李四 / Li Si"
    }
  ]
}
```

### 历史数据迁移

上线时对已有报告（有附件=已提交过的）执行一次性初始化：
- 审核状态 = `已通过`
- 审核日期 = 上传日期（I列）
- 审核人 = 责任人（L列）—— 兜底值
- 退回原因 = 空

## 页面设计

### 1. 主管审核页面（新建）

- 文件名：`FailureReport_Review.html` + `FailureReport_Review-js.html`
- 入口：首页导航栏新增「故障报告审核 / Report Review」
- 权限控制：**前端方案** — 入口对所有用户可见，页面加载时调用后端检查权限，无权限者显示提示信息

#### 后端查询函数

新增 `getPendingReviewReports()`：
- 查询 Failure_Database，筛选审核状态 = `主管审核中` 或 `已退回`
- 管理员：返回所有主管审核中/已退回报告
- 主管：返回直线上级匹配当前用户的报告（userID 表 BI 列 = 当前用户邮箱 → 该行 NAME 列 = 报告责任人）
- 返回字段：报告编号、机台号、问题描述、工序、责任人、提交日期、审核状态、内容（JSON，用于详情弹窗）

#### 页面布局
- 表格列：报告编号、机台号、问题描述、责任人、提交日期、审核状态
- 筛选：按工序（IM/TF/PK）
- **默认展示**：主管审核中 + 已退回 的报告（已通过及后续状态在 Progress 页面展示）
- 操作列：
  - `查看详情 / View Detail` — 弹窗展示完整 BDA 表单内容（只读），复用 Template 表单组件
  - `通过 / Approve` — **SweetAlert 二次确认** → 确认通过
  - `退回 / Return` — **SweetAlert 弹窗**填写退回原因（非必填）→ 退回给责任人
- 操作按钮样式：`display:block;width:120px;margin:2px auto;`（同 Upload 页面风格）
- 状态列颜色：
  - 主管审核中 = 黄色
  - 已通过 = 绿色
  - 已退回 = 红色

### 2. Upload 页面改造

- 列表范围扩展：现有 `uploadDate` 为空的报告 + 审核状态 = `已退回` 的报告
- 两种状态显示不同标记：
  - "未提交" — uploadDate 为空且审核状态 ≠ 已退回
  - "已退回"（红色标记）— 显示退回原因
- 已退回的报告：「填写故障报告」按钮变为「修改 / Edit」，点击进入 Template 编辑模式
- **Template 回填**：URL 参数传入 `&mode=edit`，页面加载时从 Failure_Database K 列 JSON 读取表单数据回填；重新提交后更新 JSON（追加 reviewHistory），审核状态回到「主管审核中」，清空退回原因列

### 3. Progress 页面改造

- **审核未通过的报告不进入 Progress 页面**（主管审核中、已退回均不显示）
- 仅审核状态 = `已通过` 的报告出现在 Progress 列表
- 现有派生逻辑不变（有附件+跟进全过→已完成）

## 后端逻辑

### 提交逻辑改造 (submitFailureReport)

```
修改前：
  保存JSON → 生成PDF → 更新附件+上传日期 → 创建跟进 → 通知PA责任人

修改后：
  保存JSON（含 reviewHistory） → 写入审核状态=主管审核中
  → 查询 userID 表获取直线上级（BI列）
  → 无直线上级：通知管理员处理
  → 有直线上级：通知主管审核（邮件抄送责任人）
  （PDF 生成 + 跟进创建 移至审核通过时执行）
```

### 审核通过逻辑 (approveReport)

```
  更新审核状态=已通过 → 更新审核人+审核日期
  → 追加 reviewHistory 到 K 列 JSON
  → 生成PDF → 更新附件+上传日期
  → 创建PA跟进记录（状态=待验证）
  → 邮件通知PA责任人（抄送主管、验证人）
```

### 审核退回逻辑 (returnReport)

```
  更新审核状态=已退回 → 更新审核人+审核日期+退回原因
  → 追加 reviewHistory 到 K 列 JSON
  → 邮件通知责任人修改（含退回原因，抄送主管）
  → 报告重新出现在 Upload 列表
```

### PDF 生成失败容错

审核通过时若 PDF 生成失败（Drive API 异常等）：
- 审核状态仍保持「已通过」
- 附件列（J列）留空
- Upload/Progress 页面该报告显示"PDF 生成失败，请重试"
- 提供「重新生成 PDF」按钮，手动触发 PDF 生成流程

## 状态流转

### 报告级别状态

```
主管审核中 / Under Review      — 提交后等待主管审核
已通过 / Approved             — 主管审核通过（进入跟进流程）
已退回 / Returned             — 主管退回修改
已上传 / Uploaded             — PDF已生成，跟进进行中（同现有）
已完成 / Completed            — 所有跟进项已通过（同现有）
```

### 报告状态机

```
                        ┌── 通过 ──→ 已通过(生成PDF+跟进) ──→ 跟进流程...
主管审核中 ──┤
                        └── 退回 ──→ 已退回 ──→ 责任人修改 ──→ 重新提交 ──→ 主管审核中
```

### Progress 页面可见性

```
审核状态 = 主管审核中 / 已退回  →  不显示
审核状态 = 已通过           →  显示（按现有派生逻辑：无附件→未上传，有附件→已上传/已完成）
```

## 通知邮件

### 邮件模板

| 场景 | 收件人 | 抄送 | 主题 | 内容要点 |
|---|---|---|---|---|
| 提交审核通知 | 主管 | 责任人 | `[EDS] 故障报告主管审核中 FR2026XXXX` | 报告编号、机台号、问题描述、责任人、提交日期 |
| 审核通过通知 | PA责任人 | 主管、验证人 | `[EDS] 故障报告审核通过 FR2026XXXX` | 报告编号、PA 内容、计划完成日期 |
| 退回修改通知 | 责任人 | 主管 | `[EDS] 故障报告审核未通过 FR2026XXXX` | 报告编号、退回原因、提示登录系统修改 |
| 无主管通知 | 管理员 | — | `[EDS] 责任人无直线上级 FR2026XXXX` | 报告编号、责任人、提示手动处理 |

> 邮件中不放审核页面链接，用户通过导航栏进入。

## 权限/角色

| 角色 | 权限 |
|---|---|
| 主管 | 审核其下属的主管审核中报告（userID BI 列匹配） |
| 责任人 | 查看自己被退回的报告，修改并重新提交 |
| 管理员 | 审核所有报告 + 处理无主管的报告 |
| 验证人/PA责任人 | 仅在审核通过后参与后续流程 |

### 主管身份判定
- userID 表中 `直线上级邮件` 列（BI 列，索引 60）
- 当前登录用户的邮箱出现在该列中 → 该用户是主管，可审核对应下属
- 单个主管（BI 列存单个邮箱）

### 导航入口可见性（前端方案）
- 审核页面入口对所有用户可见
- 页面加载时调用后端权限检查
- 无权限者：页面显示 "您暂无审核权限 / No review permission" 提示

## 边界条件

1. **退回后修改**：责任人只能修改自己提交的、状态为「已退回」的报告
2. **重复提交**：已审核通过的报告不可再修改（同现有逻辑：有附件=不可修改）
3. **审核中修改**：主管审核期间（状态=主管审核中），责任人不可修改报告
4. **审核超时**：暂不设超时，后续迭代可加提醒
5. **无主管情况**：提交时若责任人无直线上级 → 邮件通知管理员手动处理（不自动通过）
6. **Template 回填**：退回修改时从 K 列 JSON 读取表单数据回填，重新提交追加 reviewHistory
7. **修改历史**：审核记录追加到 JSON 的 `reviewHistory` 数组，不覆盖，便于追溯
8. **退回重提后审核人不变**：重新提交后直线上级不变，审核主管为同一人
9. **管理员代审**：管理员审核非自己下属的报告时，审核人字段记录管理员本人
10. **退回次数不限**：允许无限次退回-修改循环，第一版不设限制

## 交互细节

- **通过操作**：点击「通过」→ SweetAlert 确认弹窗 "确认通过此报告？" → 确认后执行
- **退回操作**：点击「退回」→ SweetAlert 输入弹窗（textarea，非必填）→ 确认后执行
- **查看详情**：弹窗展示 BDA 表单（只读模式），复用 Template 页面表单组件

## 验收标准

- [ ] 责任人提交报告后，状态为「主管审核中」，PDF 未生成，跟进未创建
- [ ] 主管收到审核通知邮件（抄送责任人）
- [ ] 主管在审核页面看到主管审核中报告列表（按工序可筛选）
- [ ] 主管可查看报告详情（BDA 只读弹窗）
- [ ] 主管可通过（二次确认）/ 退回（可填退回原因）
- [ ] 通过后自动生成 PDF + 创建 PA 跟进，通知 PA 责任人（抄送主管）
- [ ] 退回后报告回到 Upload 列表，责任人可修改重提，退回原因可见
- [ ] 退回后责任人收到通知邮件（含退回原因，抄送主管）
- [ ] 审核未通过的报告不显示在 Progress 页面
- [ ] 无直线上级时通知管理员
- [ ] 历史报告自动迁移为「已通过」状态
- [ ] 审核记录写入 JSON reviewHistory，可追溯
- [ ] 导航入口前端方案：入口常显，无权限者进入后看到提示
